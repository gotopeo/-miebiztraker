# 更新通知機能 設計ドキュメント

**作成日**: 2026年1月27日

---

## 概要

既存案件の重要な変更を検出し、ユーザーに通知する機能を実装します。新規案件通知に加えて、既に公開されている案件の締切日延長や予定価格変更などの重要な変更をユーザーに通知することで、入札機会の見逃しを防ぎます。

---

## 重要な変更項目の定義

以下の項目の変更を「重要な変更」として検出し、通知対象とします:

### 1. 締切日（applicationDeadline）
- **重要度**: 最高
- **理由**: 締切日の延長は入札参加の機会が増えるため、最も重要
- **通知条件**: 変更があった場合は必ず通知

### 2. 開札日（openingDate）
- **重要度**: 高
- **理由**: 開札日の変更はスケジュール調整が必要
- **通知条件**: 変更があった場合は必ず通知

### 3. 予定価格（estimatedPrice）
- **重要度**: 高
- **理由**: 予定価格の変更は入札戦略に影響
- **通知条件**: 変更があった場合は必ず通知

### 4. 最低制限価格（minimumPrice）
- **重要度**: 高
- **理由**: 最低制限価格の変更は入札価格に影響
- **通知条件**: 変更があった場合は必ず通知

### 5. 案件名（title）
- **重要度**: 中
- **理由**: 案件名の変更は内容の変更を示唆
- **通知条件**: 大幅な変更があった場合のみ通知

### 6. 工事場所（location）
- **重要度**: 中
- **理由**: 工事場所の変更は施工計画に影響
- **通知条件**: 変更があった場合は通知

### 7. 工期（constructionPeriod）
- **重要度**: 中
- **理由**: 工期の変更は施工計画に影響
- **通知条件**: 変更があった場合は通知

### 8. 格付（rating）
- **重要度**: 中
- **理由**: 格付の変更は入札資格に影響
- **通知条件**: 変更があった場合は通知

---

## 更新通知の送信条件

### 基本条件
1. ユーザーが更新通知を有効にしている（`enable_update_notification = true`）
2. 案件がユーザーの通知設定条件に一致している
3. 重要な変更が検出された
4. 同じ案件・同じ変更内容で既に通知済みでない

### 除外条件
1. 案件が初回観測から24時間以内（新規案件として通知済みのため）
2. 案件のステータスが「終了」または「中止」

## データベーススキーマの変更

### notification_subscriptionsテーブル
```sql
ALTER TABLE notification_subscriptions
ADD COLUMN enable_update_notification BOOLEAN DEFAULT false;
```

### notification_logsテーブル
- `notification_type`カラムに`UPDATE`を追加（既存: `NEW`）
- `message_content`カラムに変更内容を記録

## 実装の流れ

### 1. スクレイピング時
```
スクレイピング
  ↓
案件同一性判定（tender_canonical_id）
  ↓
既存案件か？
  ├─ YES → 差分判定（detectChanges）
  │         ↓
  │       重要な変更あり？
  │         ├─ YES → 更新通知対象としてマーク
  │         └─ NO  → 通常更新
  │
  └─ NO  → 新規案件として保存
```

### 2. 通知ジョブ実行時
```
通知ジョブ開始
  ↓
有効な通知設定を取得
  ↓
各通知設定について:
  ├─ 新規案件を検索 → 新規通知送信
  │
  └─ 更新通知が有効？
      ├─ YES → 更新案件を検索
      │         ↓
      │       重要な変更あり？
      │         ├─ YES → 更新通知送信
      │         └─ NO  → スキップ
      │
      └─ NO  → スキップ
```

## 重複防止ロジック

### 新規通知との重複防止
- 初回観測から24時間以内の案件は更新通知を送信しない
- 理由: 新規案件として既に通知済みのため

### 更新通知の重複防止
- `notification_logs`テーブルに変更内容のハッシュ値を記録
- 同じ案件・同じ変更内容で既に通知済みの場合はスキップ

## ユーザーインターフェース

### 通知設定フォーム
```
[ ] 更新通知を受け取る
    既に通知した案件の重要な変更（締切日延長、予定価格変更など）も通知します
```

### 通知履歴
```
種別: 更新通知
案件名: 〇〇道路改修工事
変更内容: 締切日が延長されました
通知日時: 2026-01-27 10:00
```

## テスト計画

### ユニットテスト
1. `detectImportantChanges()`関数のテスト
   - 締切日変更の検出
   - 予定価格変更の検出
   - 複数項目の同時変更の検出
   - 重要でない変更の除外

2. 更新通知メッセージ生成のテスト
   - 変更内容のフォーマット
   - 複数変更のフォーマット

### 統合テスト
1. スクレイピング → 変更検出 → 更新通知の一連のフロー
2. 重複防止ロジックの動作確認
3. 新規通知との共存確認

---

## 実装の優先順位

### Phase 1（最優先）
- 締切日変更の検出と通知
- 予定価格変更の検出と通知

### Phase 2
- 開札日変更の検出と通知
- 最低制限価格変更の検出と通知

### Phase 3
- その他の項目（案件名、工事場所、工期、格付）の変更検出と通知

---

## 注意事項

1. **通知頻度の制御**: 同じ案件で短時間に複数回変更があった場合、通知を統合する
2. **パフォーマンス**: 更新案件の検索クエリを最適化し、DBへの負荷を最小限にする
3. **ユーザー体験**: 更新通知はオプトインとし、デフォルトはOFFにする
4. **通知内容の明確化**: 何が変更されたのか一目で分かるようにする

---

## 期待される効果

1. **入札機会の増加**: 締切日延長を見逃さず、入札参加機会が増える
2. **入札戦略の最適化**: 予定価格変更を把握し、入札価格を調整できる
3. **スケジュール管理の改善**: 開札日変更を把握し、スケジュール調整ができる
4. **ユーザー満足度の向上**: 重要な変更を見逃さず、安心して利用できる
